«Тестирование программного обеспечения»

Тема: Технологии разработки программного обеспечения: "Разработка через тестирование"

Объект тестирования: Веб-приложение «Task Manager» (Менеджер задач).

1. Поиск багов и изучение библиотек тестирования (TDD)

Задание: Найти произвольное приложение и задокументировать найденные баги. Изучить инструменты модульного тестирования.

Теоретическая справка:

В рамках парадигмы TDD (Test-Driven Development) применяются инструменты автоматизации. NUnit / xUnit — популярные среды модульного тестирования. Они позволяют описывать тесты с помощью атрибутов и проверять ожидаемые результаты через утверждения (Assert). Разработка начинается с написания падающего теста, затем пишется минимальный код для его прохождения, после чего проводится рефакторинг.

2. Интеграционное тестирование

Выбранный метод: Нисходящее интеграционное тестирование (Top-down integration testing).

Ход выполнения: Тестирование архитектуры приложения проводилось сверху вниз. Проверялся модуль пользовательского интерфейса (UI React) и его взаимодействие с REST API контроллерами. Поскольку база данных (PostgreSQL) изолировалась для чистоты тестов, вместо нее использовался Mock-сервер, который возвращал заранее заготовленные JSON-ответы. Это позволило проверить корректность маппинга данных и работу сетевого слоя приложения (Axios) при статусах 200 OK, 400 Bad Request и 500 Internal Server Error.

3. Системное тестирование

Системное тестирование проводилось в staging-окружении, максимально приближенном к production:

  • Функциональное тестирование: Проверка End-to-End (E2E) сценариев с помощью инструмента Cypress. Полный цикл: регистрация -> создание проекта -> добавление задачи -> назначение исполнителя -> смена статуса на "Done".
  • Тестирование производительности (Load Testing): С помощью JMeter сгенерирована нагрузка в 500 одновременных пользователей. Выявлено, что среднее время ответа (Response Time) эндпоинта /api/tasks/search возрастает с 120мс до 4.5 секунд при поиске по частичному совпадению текста.
  • UI/UX тестирование адаптивности: На устройствах с шириной экрана менее 375px (iPhone SE) боковое меню наезжало на контент, блокируя кнопку "Настройки".

4. Ручное тестирование, составление тестовых случаев

Тестовый случай (Test Case) #TC-014: Ограничение размера прикрепляемого файла

Параметр Значение
Цель: Убедиться, что система отклоняет файлы размером более 10 МБ и выводит корректное сообщение.
Предусловия: Пользователь авторизован. Подготовлен тестовый файл test_archive.zip размером 12 МБ.
Шаги: 1. Открыть любую существующую задачу.
2. Нажать на иконку "Скрепка" (Прикрепить файл).
3. Выбрать файл test_archive.zip (12 МБ) в диалоговом окне.
4. Дождаться окончания загрузки.
Ожидаемый результат: Загрузка прерывается. Индикатор прогресса окрашивается в красный. Появляется всплывающее уведомление (Toast): "Ошибка: Размер файла превышает максимально допустимый (10 МБ)".
Фактический результат: Система пытается загрузить файл, через 30 секунд падает с ошибкой 413 Payload Too Large. Уведомление для пользователя отсутствует (бесконечный лоадер).
Статус: FAILED (Провален)
