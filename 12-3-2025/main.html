Проект «Корпоративная доска заметок»

Концепция проекта, состав работ и анализ рисков для внедрения корпоративной доски заметок (ToDo Board) в четырёх подразделениях компании. Современное решение для управления задачами 200+ сотрудников.

Автор: Головкин Илья, 9/3 РПО 23/2 Дата: 12/3/2025

    1. Устав проекта

    1.1. Название проекта

    «Корпоративная доска заметок (ToDo Board) для 4 подразделений компании»

    • Заказчик: []Основной стейкхолдер проекта, ответственный за финансирование и приёмку
    • Инициатор проекта: []Инициатор и бизнес-спонсор проекта

    1.3. Цели проекта

    • Обеспечить сотрудников (до 200 человек) единой доской для планирования и отслеживания задач.
    • Сократить время на согласование задач и напоминания за счёт автоматизации уведомлений (Outlook, SMS).
    • Обеспечить масштабируемость решения для дальнейшего выхода на мобильные и web-клиенты.

    1.4. Область охвата (Scope)

    В проект входят:

    • Разработка десктопного приложения на WPF:
      • создание, редактирование и удаление стикеров;
      • произвольное размещение и перетаскивание по экрану;
      • цветовая дифференциация и нестандартные формы стикеров;
      • передача стикера пользователю в рамках своего подразделения.
    • Разработка REST API/сервиса для работы с данными и интеграций.
    • Хранение данных в MS SQL Server на выделенном сервере.
    • Локальная XML-сериализация заметок пользователя (личный backup).
    • Интеграция с Outlook для установки и контроля сроков задач.
    • Интеграция с SMS-шлюзом для рассылки уведомлений.
    • Подготовка и настройка серверов БД и приложений (SMS, конфигурация, защита).

    1.9. Критерии успеха проекта

    • Система используется во всех 4 подразделениях в течение не менее 1 месяца без критических инцидентов.
    • Среднее время реакции на задачу сократилось на X% (по сравнению с «до»).
    • Система поддерживает до 200 одновременных пользователей без заметной деградации производительности.
    • Уровень удовлетворённости пользователей по результатам опроса — не ниже Y/5.

    • Нехватка ключевых специалистов в нужные моменты проекта.

    Раздел 3 · Прогноз рисков

    3. Прогноз рисков

    Риски отсортированы по убыванию «тяжести» (Impact × Probability). Для ключевых рисков определены базовые меры реагирования.

    3.1. Матрица рисков проекта

    № Риск Вероятность Влияние Тяжесть / приоритет Меры реагирования
    1 Срыв интеграции с Outlook (ограничения API, политика безопасности) Средняя Высокое Очень высокий Раннее прототипирование интеграции, согласование с ИТ-службой, запас по срокам, альтернативный сценарий.
    2 Уязвимая конфигурация сервера/БД → утечка данных Низкая–ср. Критическое Очень высокий Политики безопасности, аудит прав, шифрование каналов, обновления, тестирование безопасности.
    3 Нестабильность SMS-шлюза / задержки уведомлений Средняя Среднее–выс. Высокий Надёжный провайдер, логирование статусов, дублирование уведомлений через Outlook/почту.
    4 Существенные изменения требований после начала разработки Средняя Высокое Высокий Процесс change-request, фазовая поставка (MVP), прототипы и ранние демо.
    5 Недоступность ключевых специалистов (болезнь, увольнение) Средняя Среднее Средний–высокий Резервирование задач, обмен знаниями (код-ревью, документация), перекрытие компетенций.
    6 Недооценка нагрузки (200 одновременных пользователей) Средняя Среднее Средний Нагрузочное тестирование, мониторинг, запас по ресурсам, оптимизация запросов к БД.
    7 Низкое пользовательское принятие (неудобный UX) Средняя Среднее Средний Вовлечение пользователей в UAT, пилотный запуск, сбор обратной связи и доработка интерфейса.
    8 Задержки с поставкой оборудования и лицензий Низкая Среднее Низкий–средний Ранний заказ, временное использование виртуальных машин или облачных ресурсов.
    9 Ошибки в локальной XML-сериализации → потеря личных backup-файлов Низкая Среднее Низкий–средний Тестирование сериализации, рекомендации по периодическим серверным backup-ам, информирование пользователей.

    3.2. Общий подход к управлению рисками

    • Регулярное обновление реестра рисков (на статус-встречах проекта).
    • Назначение ответственных за ключевые риски.
    • Планирование превентивных действий и процедур реагирования (contingency plan).
    • Фиксация изменений статусов рисков в отчётности по проекту.
